# Sandbox 性能测试报告

基于 `main.py` 跑 **100 次** `Sandbox.create()` + `run_code()` 完整生命周期统计。

## 测试环境

| 项 | 值 |
|---|---|
| template | `code-code-vpc` |
| 沙箱 timeout | 3600s |
| run_code 超时 | 60s |
| 网关 | `E2B_DOMAIN=ap-shanghai.tencentags.com`（腾讯 AGS e2b 兼容网关） |
| 测试负载 | `print('hello python')` |

每次循环做的事：

```python
start = time.time()
sandbox = Sandbox.create(template="code-code-vpc", timeout=3600)
result = sandbox.run_code("print('hello python')", timeout=60)
elapsed = time.time() - start
print(f"[{i}/{RUNS}] Lifecycle: {elapsed:.2f}s (create: {create_elapsed:.2f}s, run_code: {run_elapsed:.2f}s)")
sandbox.kill()
```

> 计时只覆盖 `create + run_code`，不含 `kill()`。

## 总体统计 (n=100, 全部成功)

| 指标 | min | p50 | p90 | p95 | p99 | max | mean | σ |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| Lifecycle | 0.10 | 0.13 | 0.17 | 0.25 | 0.54 | 0.54 | 0.15 | 0.07 |
| create | 0.05 | 0.07 | 0.10 | 0.12 | 0.14 | 0.14 | 0.08 | 0.02 |
| run_code | 0.05 | 0.06 | 0.08 | 0.13 | 0.41 | 0.41 | 0.07 | 0.06 |

所有 100 次 `stderr` 为空、`Error: None`。

## 核心结论

### 1. create 已经稳定，不是双峰分布了

`create` 分布：

| 区间 (s) | 次数 | 直方 |
|---|---:|---|
| [0.00, 0.07) | 29 | `####################################################################################################################` |
| [0.07, 0.10) | 60 | `################################################################################################################################################################################################################################################` |
| [0.10, 0.13) | 8 | `################################` |
| [0.13, 0.20) | 3 | `############` |

**没有慢簇**：100 次里没有任何一次 `create > 0.20s`。上一次跑出现的 0.30–0.48s 慢簇这次完全消失，说明 AGS 预热池已被该模板填满。

### 2. run_code 仍有一个 3 次的右尾

`run_code` 分布：

| 区间 (s) | 次数 | 直方 |
|---|---:|---|
| [0.00, 0.06) | 43 | `####...####` |
| [0.06, 0.10) | 49 | `####...####` |
| [0.10, 0.15) | 4 | `################` |
| [0.15, 0.20) | 1 | `####` |
| [0.30, 0.50) | 3 | `############` |

49 次（49%）稳态在 0.06-0.10s，4 次明显异常跑到 0.30-0.41s，**`create` 都很快**（0.05-0.13s），说明这 3 次不是冷启动路径，是沙箱内部 Python 解释器第一次启动的开销（AGS 接到请求后起 Python 子进程）。

### 3. create ↔ run_code 相关性下降

| 指标 | 上次 | 本次 |
|---|---:|---:|
| 相关系数 | 0.720 | 0.527 |

含义：上次的"create 慢 ⟹ run_code 慢"耦合，这次被打破。`create` 已稳定，所以 `run_code` 的尾延迟完全来自沙箱自身。

## 对比上次运行

| 指标 | 上次 | 本次 | 变化 |
|---|---:|---:|---:|
| Lifecycle p50 | 0.35 | 0.13 | **−0.22** |
| Lifecycle p95 | 0.93 | 0.25 | **−0.68** |
| Lifecycle max | 1.02 | 0.54 | **−0.48** |

**降幅 ~70%**。验证了上一份报告里的猜测——上一次测确实是 AGS 模板冷启动之后第一次拿到的延迟画像，本次的才是稳态。

## 按 create 阈值切两簇对比

以 `create = 0.10s` 为界：

| 簇 | 占比 | lifecycle 均值 | run_code 均值 |
|---|---:|---:|---:|
| 预热池命中 | 94% | 0.13s | 0.06s |
| 实际调度 | 6% | 0.37s | 0.24s |

慢簇从去年的 ~52% 降到 6%，慢簇内部的实际延迟也从 0.30–0.48s 降到 0.30s 量级。

## 关键结论与建议

- **稳态下 p50 = 0.13s、p95 = 0.25s**——这套网关在预热充分时性能可用。
- **create 已经稳定**：瓶颈现在不在沙箱调度，在沙箱内部 Python 子进程冷启动（每次新代码要起解释器）。
- **`run_code` 的 p99 出现 0.41s 单峰**：是 Python 子进程启动开销，不是网络问题。如果场景允许，可考虑在 `python_code` 里加 `from xxx import yyy` 等让解释器复用常驻（取决于 AGS 是否支持 keep-alive 解释器）。
- **对外 SLA 仍建议按 p95 报**：p99 / max 还有 0.41-0.54s 毛刺，p50 已不能反映尾延迟体感。

## 附：同步/异步

`Sandbox.create()` 是 **同步** 调用。`e2b_code_interpreter.Sandbox` 继承自 `e2b.sandbox_sync.Sandbox`，整条调用链（`requests` / HTTPX 同步客户端）会阻塞当前线程。

100 次串行总耗时 ~ 15s（本次均值 0.15s × 100）。想压榨更高吞吐，可走：

- 切换到 `AsyncSandbox` + `asyncio.gather()` 并发
- 多进程 / 多线程并行触发
